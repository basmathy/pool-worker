# pool-worker

Проект разделен на пакеты:

- `ru.basmathy.pool.api` - публичный интерфейс `CustomExecutor`.
- `ru.basmathy.pool.config` - immutable-конфигурация `PoolConfig`.
- `ru.basmathy.pool.core` - `CustomThreadPool`, `PoolWorker` и простой `PoolLogger`.
- `ru.basmathy.pool.factory` - `NamedThreadFactory`.
- `ru.basmathy.pool.routing` - маршрутизация задач через `TaskRouter` и `RoundRobinTaskRouter`.
- `ru.basmathy.pool.rejection` - политика отказа `RejectPolicy` и `AbortRejectPolicy`.
- `ru.basmathy.pool.task` - демонстрационная задача `DemoTask`.

`CustomThreadPool` хранит список рабочих потоков. У каждого `PoolWorker` есть собственная `ArrayBlockingQueue<Runnable>`, поэтому в проекте нет общей очереди задач. Пул принимает задачи через `execute` и `submit`; `submit` оборачивает `Callable` в `FutureTask`.

## Round Robin (алгоритм балансировки)

`RoundRobinTaskRouter` выбирает стартовую очередь по кругу. Если выбранная очередь заполнена, пул проверяет остальные очереди. Если свободного места нет, но количество рабочих потоков меньше `maxPoolSize`, пул создает новый рабочий поток и кладет задачу в его очередь. Если и это невозможно, вызывается политика отказа.

## AbortRejectPolicy

`AbortRejectPolicy` логирует отказ и выбрасывает `RejectedExecutionException` - так вызывающий код сразу узнает, что задача не была принята.

Недостатки:

- вызывающий код обязан обрабатывать исключение;
- задачи не накапливаются сверх заданного размера очередей;
- при кратковременных всплесках нагрузки часть задач может быть потеряна, если отправитель не делает повторную попытку.

## Сравнение с ThreadPoolExecutor

`ThreadPoolExecutor` из JDK более универсален: он поддерживает разные очереди, политики отказа, точки расширения жизненного цикла, статистику и хорошо отлаженную конкурентную реализацию. Этот проект проще, но зато прозрачнее для обучения: рабочие потоки, очереди, распределение задач и завершение работы реализованы явно.

Главное отличие этой реализации — отдельная очередь на каждый `PoolWorker`. У стандартного `ThreadPoolExecutor` обычно используется общая `BlockingQueue`, из которой рабочие потоки забирают задачи.

## Мини-исследование параметров

- `corePoolSize` задает базовое число рабочих потоков. Чем оно выше, тем меньше задержка при начальном запуске, но тем больше постоянное потребление ресурсов.
- `maxPoolSize` ограничивает рост пула при перегрузке. Большее значение помогает переживать пики нагрузки, но может привести к лишним переключениям между потоками.
- `queueSize` задает емкость каждой очереди `PoolWorker`. Большие очереди уменьшают число отказов, но увеличивают время ожидания задач.
- `keepAliveTime` определяет, как долго лишний рабочий поток ждет новую задачу. Малое значение быстрее освобождает ресурсы, большое уменьшает частое создание новых потоков.
- `minSpareThreads` поддерживает минимальное число свободных рабочих потоков, если пул еще может расти. Это снижает задержку приема следующих задач, но может временно держать больше потоков.
